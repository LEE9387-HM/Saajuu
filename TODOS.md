## P2 — PWA 홈화면 추가

- **What**: manifest + (필요 시) 최소 서비스워커로 홈화면 설치 지원
- **Why**: 일일 운세와 결합해 "매일 여는 앱" 경험 완성
- **Pros**: 재방문 마찰 제거, 웹 푸시의 전제 조건
- **Cons**: 카톡 인앱 브라우저 설치 불가·iOS 프롬프트 API 없음 — 딥링크 유입 대부분에 무효. SW 캐시는 stale 버전 서빙 리스크
- **Context**: 외부 검증에서 "유입 채널과 상충, 지금은 리스크만 얹는다" 판정. v0.4.0 배포 후 GoatCounter의 재방문율 데이터가 유의미해지면 manifest만 먼저 추가(SW 없이). SW를 넣을 땐 network-first + 버전 스탬프 캐시명 필수. GitHub Pages 하위 경로(`/Saajuu/`)에서 scope·start_url 상대 경로 주의
- **Blocked by**: v0.4.0 배포 + 재방문 데이터 확보

## P2 — 띠별 운세 SEO 정적 페이지

- **What**: 12띠 × 정적 페이지 (검색 유입 + 애드센스 심사용 콘텐츠 볼륨)
- **Why**: 무료 트래픽 확보. 수익화 Phase 0의 트래픽 과제와 직결
- **Pros**: 일진 엔진 재사용으로 콘텐츠 원가 낮음
- **Cons**: 애드센스는 github.io 서브도메인 등록 불가 — 커스텀 도메인 선행. 크롤러는 JS 렌더 콘텐츠를 못 보므로 빌드타임 생성 + 매일 GitHub Actions cron 재빌드 필요. 단일 SPA → 멀티페이지 Vite 설정 전환 비용. "띠별 운세"는 초경쟁 키워드
- **Context**: 착수 조건 = ① 커스텀 도메인 구매(연 ~2만 원) ② `.github/workflows`에 daily cron 빌드 추가 설계 ③ 사이트맵·메타 정비. 롱테일 키워드(예: "병오년 말띠 7월 운세")로 시작하는 편이 현실적
- **Blocked by**: 커스텀 도메인 구매 결정

## P3 — 다중 프로필 (가족·지인 사주)

- **What**: localStorage 프로필을 단일 → 복수(이름 라벨 지정)로 확장
- **Why**: 사주 앱 실사용 패턴은 "가족·지인 사주 대신 봐주기" — 단일 프로필은 남의 사주 한 번 보면 내 프로필이 덮이는 구조
- **Pros**: 리텐션 강화, 유료 궁합 상품(2인 입력)의 자연스러운 기반
- **Cons**: 프로필 선택 UI·마이그레이션 복잡도
- **Context**: v0.4.0의 프로필 스키마에 `{ v: 1 }` 버전 필드를 넣어두었으므로 마이그레이션 경로는 준비됨. GoatCounter 데이터에서 "정보 수정" 빈도가 높게 나오면 착수 신호
- **Blocked by**: v0.4.0 프로필 사용 데이터
